# Task Flow Rules

所有角色共享的任务流转规则。角色 Skill 只写自己的职责，状态流转、归档、派生、审查分流和对用户汇报口径以本文件为准。

## 主文件

| 文件 | 用途 |
|---|---|
| `ai/TASK_BOARD.md` | 当前执行索引，只保留非 `DONE` 任务 |
| `ai/TASK_LOCKS.md` | 当前任务占用表，用于多个 flow 并行时避免重复领取 |
| `ai/TASK_MODULES.md` | 任务模块目录和模块优先级口径 |
| `ai/TASK_ARCHIVE.md` | 已 `DONE` 任务归档索引 |
| `ai/PM_DECISION_LIST.md` | 当前待用户拍板的项目经理决策项 |
| `ai/PM_DECISION_ARCHIVE.md` | 已完成的项目经理决策项 |
| `.ai/tasks/<TaskId>/flow-context.md` | flow 针对单个任务的运行上下文和调度依据 |

## 新建任务责任边界

- 新建业务任务、创建 `.ai/tasks/<TaskId>/` 目录、写 `task.md`、写初始 `handoff.md`、登记 `TASK_BOARD.md` 和完成入场检查，只能由项目经理负责；flow 不直接创建业务任务或任务文件夹。
- Owner 为 `aiskill` 的协作资产任务例外：用户要求修改协作资产时，由 aiskill 按同一任务载体要求创建 `.ai/tasks/<TaskId>/`、`task.md`、初始 `handoff.md` 和任务板索引或完成归档记录；完成自检后直接归档到 `TASK_ARCHIVE.md`，不进入测试或审查队列。
- flow 发现审查问题、测试失败、用户补充或阻塞需要新建承接任务时，只记录当前任务的 `flow-context.md` 并切项目经理处理；项目经理判断是否创建、关闭旧任务、设置依赖门禁、暂停下游和指定下一处理身份。
- 项目经理完成新任务入场检查后，把控制权交还 flow；交接里的 `下一步处理身份` 必须写具体角色或 `无`，`下一步动作` 写“控制权交还 flow 继续扫描/调度”。
- 项目经理不创建新任务的 `flow-context.md`。新任务的 `flow-context.md` 由 flow 在接回控制权后第一次实际读取、校准、领取、释放、跳过或阻塞该任务时创建并维护。

## 状态流转

```text
BACKLOG -> PLANNED -> ASSIGNED -> IN_PROGRESS -> CODE_READY -> TESTING -> REVIEW_READY -> REVIEWING -> DONE
TEST_FAILED -> 项目经理裁决 -> FIX_REQUIRED -> IN_PROGRESS -> CODE_READY -> TESTING -> REVIEW_READY
REVIEWING -> DONE + 分类问题清单 -> 项目经理新增承接任务 / 阻断承接任务 / PM 决策
任意执行状态 -> BLOCKED -> 项目经理决策 -> ASSIGNED / PLANNED / DONE
```

## 下一步处理身份

- 状态只记录任务所处阶段；flow 调度优先看最近交接中的 `下一步处理身份`。当任务为 `REVIEW_READY` / `REVIEWING` 且下一步处理身份为审查时，flow 可以受控切换到审查身份执行最终审查。
- `flow` 是调度控制权，不是任务处理身份；任何任务的 `当前处理身份`、交接中的 `下一步处理身份` 都不得填写 `flow`。需要交还 flow 时，写在 `下一步动作` 或 `flow-context.md`，身份字段必须是具体角色或 `无`。
- 角色完成工作时，必须在交接中写简短 `本次完成`、`状态变更`、`下一步处理身份`、`下一步动作` 和 `验证/未验证`；项目经理仅“立任务”时不强制写 `本次完成` 1/2/3。
- 项目经理处理 `TEST_FAILED`、`PM_REVIEW`、`BLOCKED`、`ACCEPTED`、承接任务、归档和改派时，必须写明 `下一步处理身份`。
- flow 调度顺序：先检查 `PM_DECISION_LIST.md` 和 `TASK_LOCKS.md`，再读取最近交接的 `下一步处理身份`，最后才按状态归属兜底。
- `PM_DECISION_LIST.md` 中 `OPEN` 决策默认只阻塞关联任务；flow 跳过被 `OPEN` 决策阻塞的任务后，必须继续扫描其他可执行任务。
- 只有决策项明确写明 `阻塞范围：模块` 时，flow 才跳过该模块；明确写明 `阻塞范围：全局` 时，flow 才停止全局调度并切项目经理沟通。
- 如果 `下一步处理身份` 缺失，默认不猜，交项目经理补充或裁决；但当任务状态、Owner、锁、PM 决策和任务说明均无冲突时，flow 允许按状态归属兜底一次，并在交接中补写“按状态兜底”的下一步处理身份。
- 如果 `下一步处理身份` 与任务状态、Owner、锁或模块占用冲突，flow 不猜，交项目经理处理。
- 旧任务不批量迁移；本规则从项目启用当前 kit 口径后的新交接开始生效。
- 历史 `handoff.md` 缺少 `下一步处理身份` 时不批量改写；只有该任务被实际读取或调度时，flow 或项目经理按安全兜底规则补一条轻量校准交接。
- 历史交接若写 `下一步处理身份：审查` 且任务当前仍为 `CODE_READY`，按旧流程兼容为 `下一步处理身份：测试`；只有状态为 `REVIEW_READY` / `REVIEWING` 时才归最终审查。

## 主执行 Agent

- `Owner` 表示项目责任角色；`主执行 Agent` 表示本轮实际执行工具或主 Agent，两者必须分开记录。
- 新建任务必须在 `TASK_BOARD.md` 和 `.ai/tasks/<TaskId>/task.md` 的基本信息表写 `主执行 Agent`。默认由 Codex 执行时填 `codex`；交给 Cursor CLI 执行时填 `cursor`；交给 Claude Code 执行时填 `claude`；其它外部 Agent 填 `其他：名称`。
- 任务执行权发生变化时，Codex 或当前控制角色必须先更新 `主执行 Agent`，再调用外部 Agent 或继续执行；交接文件也要同步写明本轮 `主执行 Agent`。
- 复检和返工时，优先按任务的 `主执行 Agent` 找回对应 Agent：`cursor` 返给 Cursor，`claude` 返给 Claude Code，`codex` 由 Codex 或项目角色继续处理。若用户明确改派或当前问题性质变化，必须在交接里说明为何改写 `主执行 Agent`。
- 历史任务不批量补猜 `主执行 Agent`；首次被项目经理、flow 或外部 Agent 调用链实际触碰时，再按最新事实补齐。

## Runtime 行为状态机

业务任务状态与 AI 运行时行为状态分层处理。任务板状态仍以本文件为准；每个角色执行过程必须同时遵守 `.agents/context/RUNTIME_RULES.md`：

```text
PLAN -> ACT -> VERIFY -> RUNTIME_DONE
PLAN -> ACT -> VERIFY -> ACT
PLAN -> ACT -> BLOCKED
```

`RUNTIME_DONE` 只表示当前 AI 运行循环结束，不等同于任务板 `DONE`。禁止 `PLAN -> PLAN -> PLAN`、`REVIEW -> REVIEW -> REVIEW`、`THINK -> THINK -> THINK`。连续 2 次对用户输出、状态更新或可见执行步骤没有工具、文件证据、diff、命令结果或任务状态变化时，必须切换到 ACT。

- `CODE_READY` / `TESTING` 归测试。
- `TEST_FAILED` / `PM_REVIEW` / `ACCEPTED` 归项目经理验收、裁决或承接。
- `REVIEW_READY` / `REVIEWING` 归审查；审查完成后原任务进入 `DONE`。
- `FIX_REQUIRED` / `ASSIGNED` 归对应执行 Owner；`FIX_REQUIRED` 只能由项目经理裁决后进入。
- `INTEGRATED` 归项目经理归档。
- `PLANNED` / `BACKLOG` / `BLOCKED` 归项目经理收口。
- Owner 为 `aiskill` 的协作资产任务由用户和 aiskill 直接确认，自检后直接 `DONE`，不进入测试或审查队列；用户明确授权的协作机制顺序依赖链可由 flow 交给 aiskill，并在归档后继续链上下一任务。
- `DONE` 任务从 `TASK_BOARD.md` 移到 `TASK_ARCHIVE.md`，主面板不保留历史详情。

## PM_REVIEW 子类型

- `PM_REVIEW` 必须在任务备注或交接中写明子类型，避免混读测试失败和审查收口。
- 子类型只用短值：`post-review-acceptance`（审查无阻断，待项目经理验收或归档）、`post-review-followup`（审查有问题，待项目经理承接）、`post-test-fail`（测试失败后待项目经理裁决）、`legacy-review`（历史审查流转遗留）。
- flow 看到 `PM_REVIEW` 时仍交项目经理；项目经理必须先按子类型判断是验收、一次返工、阻断承接、普通承接、关闭还是 PM 决策。

## 依赖满足与测试审查并行

- 测试和最终审查不能成为 flow 阻断点；上游任务到 `CODE_READY` 后，默认视为已经产出可供下游继续的实现结果。
- 默认依赖满足状态：`CODE_READY`、`TESTING`、`REVIEW_READY`、`REVIEWING`、`PM_REVIEW`、`ACCEPTED`、`INTEGRATED`、`DONE`。
- 默认依赖未满足状态：`BACKLOG`、`PLANNED`、`ASSIGNED`、`IN_PROGRESS`、`FIX_REQUIRED`、`TEST_FAILED`、`BLOCKED`。
- 下游任务如果处于 `PLANNED`，且依赖均已满足、Owner 是项目经理/前端/后端/架构师/测试，flow 可以按锁规则直接调度，不得因为上游还在 `CODE_READY` / `TESTING` / `REVIEW_READY` / `REVIEWING` 而停住。
- 只有任务说明或任务备注明确写了“必须等测试通过”“必须等审查完成”“必须等验收完成”“依赖状态要求 `REVIEW_READY` / `DONE`”时，才把依赖门槛提高到对应状态。
- 新任务必须填写 `依赖门禁`：默认 `CODE_READY`；涉及公共契约、协议字段、生成类型、数据库结构、Swagger/proto、Topic、token 或设备密钥时，默认提高到 `REVIEW_READY` 或写明需要用户确认。
- 上游后续若被项目经理裁决为 `FIX_REQUIRED`、`BLOCKED` 或新增承接任务，项目经理负责判断是否暂停下游、返工或重建任务；不得让历史待审状态预先卡死下游 flow。

## 顺序依赖任务链自动推进

- 用户明确要求把多个任务按顺序依赖推进，或任务备注写明 `顺序依赖链` 时，flow 必须把这些任务视为同一条当前链。
- 链上任务完成、阻塞、释放锁或控制权交还 flow 后，flow 必须优先重新扫描同一条链，不重新询问优先模块，不逐任务请求确认。
- 下游任务只有在依赖门禁已满足、任务说明和交接文件存在、无 `ACTIVE` 锁冲突、无关联 `OPEN` PM 决策阻塞、允许范围清晰时，才能自动调度。
- 链上任务 Owner 为 `aiskill` 且模块为 `协作机制` 时，flow 可以在用户明确授权的链内把控制权交给 aiskill；aiskill 完成自检后仍按协作资产任务规则直接归档，不进入测试或审查队列。
- 若链上任一任务依赖未满足、范围不清、任务说明缺失、被锁占用、被 PM 决策阻塞或用户主动切换模块，flow 停在该点并说明阻塞原因。

## flow 调度排序

- flow 是任务流大管家，不只是状态机；每轮必须同时考虑模块、工作量、交付距离、任务锁、依赖和上下文预算。
- flow 不仅按状态领取任务；有最近交接时，优先按交接里的 `下一步处理身份` 调度。
- flow 先按任务 `模块` 字段聚合候选任务；用户指定模块优先级时先处理指定模块，未指定时优先选择同模块内最接近交付且工作量较小的任务。
- 已选定顺序依赖任务链时，该链优先于普通同模块排序；链上当前可执行任务完成后，flow 回来继续扫描链上下一个依赖已满足的任务。
- 同模块内优先推进差一步可交测试的任务：项目经理已裁决的 `FIX_REQUIRED`、可直接执行的 `ASSIGNED`、依赖已满足且无强门禁的 `PLANNED`。
- `CODE_READY` 任务归测试；flow 可以调度测试执行验证。
- `TEST_FAILED` / `PM_REVIEW` 任务归项目经理；flow 不得直接调度给执行角色返工。
- `REVIEW_READY` / `REVIEWING` 任务归最终审查；flow 可以受控切换到审查身份执行审查。切换前必须读取任务 `task.md`、最近 `handoff.md`、`flow-context.md`、`code-reviewer` Skill、锁表和 PM 决策清单；切换后必须完整遵守审查角色规则。
- 完成同模块低工作量任务后，flow 回来继续扫描同模块，按工作量从小到大处理第二、第三候选任务；遇到锁冲突、依赖不满足或范围不清时跳过并交对应角色。
- 项目经理收口类任务如果阻断后续执行或存在与候选任务相关的 `OPEN` 决策项，可以插队；但 flow 不替项目经理做业务决策。

## flow 启动模块摘要

- 用户说 `flow，开始工作`、`切换 flow 身份` 或同类指令时，如果没有明确指定模块，flow 必须先汇报模块摘要；当前只剩一个模块时直接开始该模块调度，存在多个模块时才等待用户选择优先模块。
- 模块摘要必须按 `TASK_BOARD.md` 当前非 `DONE` 任务统计，并固定使用 `模块 / 剩余 / 主要状态 / 可调度` 四列；不得自定义表头。
- `剩余` 填该模块当前非 `DONE` 任务数量；`主要状态` 填最关键状态组合；`可调度` 填 `是/否/部分` 和一句短原因。
- 当前涉及多个模块时，flow 提供 2-4 个优先模块选项；选项只到模块层，不展开任务长清单。
- 当前只涉及一个模块时，flow 不输出优先模块选项、不等待用户选择，直接把该模块设为当前选定模块并开始调度。
- 选项排序按：可快速交付优先、阻塞程度高优先、待测试或待审查积压明显优先、用户最近关注优先。
- 用户选定模块后，flow 再按模块内交付距离、工作量、风险和锁状态自动调度；不逐任务重复询问。
- 用户已经明确指定模块时，flow 可跳过选项，直接输出该模块状态灯并调度。
- 用户选定模块后，该模块在当前 flow 会话内持续生效；flow 每次完成任务回到扫描时，必须继续该模块内下一个可执行任务，不得重新询问优先模块。
- 只有选定模块没有 flow 可执行任务、被其他 flow `ACTIVE` 占用、被 `OPEN` 决策项以模块/任务范围全部阻塞、任务说明缺失/范围不清，或用户主动切换模块时，flow 才能停止或重新询问。

## 工作量评估

- flow 调度前必须做轻量工作量分级，不要求精确工时，只判断小/中/大/不确定。
- 小：单文件或少量同模块文件，验收清晰，无跨仓库、无协议、无公共 API 影响，可能一次最小补丁进入 `CODE_READY`。
- 中：同模块多文件、前后端类型同步或需要多项验证，但边界和验收明确。
- 大：跨模块、跨仓库、公共契约、协议字段、任务边界不清、依赖未稳定或需要重新设计。
- 不确定：任务说明缺失、允许修改范围不清、依赖或锁状态矛盾；交项目经理或架构师，不猜测领取。

## flow 上下文预算

- flow 每轮扫描只保留候选摘要：任务 ID、模块、状态、Owner、依赖、工作量级别、交付距离、锁状态和阻塞点。
- 切换执行角色前，flow 必须压缩上下文为最小执行包：任务 ID、模块、目标、允许/禁止范围、依赖状态、最近交接或审查结论、建议验证方式。
- 禁止把完整任务板、历史侦查过程、无关模块任务和旧对话长上下文带入执行角色。
- 候选任务过多、历史过长、同模块任务链复杂或连续两轮扫描无调度时，flow 必须先压缩当前选定模块队列摘要，再重新读取该模块下一个可执行任务文件执行；不得因为上下文预算不足而重新询问优先模块。

## 前三关阻塞调度

- 当前阶段为 `需求澄清`、`验收标准` 或 `方案设计`，且阶段状态为 `阻塞` 时，flow 不得推进阶段、不得按 Owner 兜底领取、不得调度到开发/测试/审查。
- 前三关阻塞任务只能归属项目经理沟通；flow 面向用户只提示存在需要拍板的语义阻塞，并按阻塞范围继续扫描其他可执行任务。
- 用户确认后，必须由项目经理或对应阶段负责人更新阶段产物和阶段状态，flow 才能继续推进。

## flow 运行上下文文件

- flow 只要实际读取、校准、领取、释放、跳过、阻塞或承接一个任务，必须在该任务目录维护 `.ai/tasks/<TaskId>/flow-context.md`；没有该文件时先创建。
- 对新建任务，`flow-context.md` 的第一条记录必须发生在项目经理完成入场检查并交还 flow 之后；记录 flow 接回依据、任务阶段、依赖/锁判断、本轮调度结论和下一步具体处理身份。
- `flow-context.md` 是 flow 自己的运行账本，不替代 `task.md`、`handoff.md`、测试报告或审查报告；它只记录 flow 为什么做出当前调度判断。
- 每次更新必须按写入时间追加一条记录，至少包含：时间、触发来源、读取依据、任务状态、当前阶段、阶段状态、当前处理身份、依赖判断、锁判断、我在做什么、下一步动作、下一步具体处理身份、阻塞原因；补录历史事实时，标题必须写明“补录”，不得插回旧时间位置造成时间顺序倒挂。
- flow 发现或调度审查承接任务前，必须在 `flow-context.md` 写清来源任务、来源报告、承接问题、依赖门禁、是否暂停下游、为何交给当前角色；缺少任一依据时，不得猜测承接关系，必须交项目经理。
- 来源任务的 `flow-context.md` 只能记录来源任务在本轮的下一步身份，例如 `项目经理` 或 `无`；新承接任务的执行角色只能写在承接任务自己的 `task.md`、`handoff.md` 或新任务 `flow-context.md` 中，禁止把承接任务下一身份写成来源任务的“下一步具体处理身份”。
- flow 发现 `当前处理身份` 或交接 `下一步处理身份` 被写成 `flow` 时，不得继续按该字段调度；必须先追加 `flow-context.md` 纠偏记录，再按任务状态、阶段流程和最近有效交接校准为具体角色，无法校准时交项目经理。
- 依赖未满足、强门禁未达到或需要等待用户拍板时，不得把 `当前阶段` 改成自造阶段；等待原因写入 `阶段状态`、任务备注或 `flow-context.md`，`当前阶段` 仍保持标准阶段。
- flow 受控切换审查前后都必须追加 `flow-context.md`：审查前记录为什么可以切审查、锁状态和读取依据；审查后记录审查结论、状态变更、是否产生承接或 PM 决策、控制权是否回到 flow。

## 并发与任务锁

- 多个 flow 可以并行启动，但必须通过 `ai/TASK_LOCKS.md` 做任务级占用。
- 当前阶段任务锁心跳不做自动化工具；flow 或执行角色在长任务开始、超过 15 分钟仍未完成、完成/阻塞/放弃前，必须人工更新或释放 `TASK_LOCKS.md`。后续接入 LangGraph 后再评估自动心跳。
- 并行 flow 必须同时遵守模块级互斥：同一模块同一时间只允许一个 flow 操作相关代码。
- flow 领取任务前，必须读取 `ai/TASK_LOCKS.md` 并判断目标任务所属模块是否已有其他 flow 的 `ACTIVE` 占用；有则跳过该模块全部任务。
- 新占用记录必须写明任务 `模块`；历史占用缺少模块字段时，flow 通过任务 ID 回查 `ai/TASK_BOARD.md` 的 `模块` 字段推断。
- 如果除被占用模块外没有其他可执行任务，当前 flow 直接结束并说明该模块已有 flow 处理中；不得打扰、催促、释放、改派或抢占另一个运行中的 flow。
- flow 领取 `ASSIGNED` / `FIX_REQUIRED` / `CODE_READY` 测试任务 / 可由项目经理收口的任务前，必须先读取 `ai/TASK_LOCKS.md`；若任务已存在其他执行者写入的 `ACTIVE` 占用则跳过。
- flow 领取任务时，先将任务状态置为 `IN_PROGRESS`，再在 `ai/TASK_LOCKS.md` 写入 `ACTIVE` 占用（必须附带自己的**执行标识**，如 `执行者: 当前对话-特定时间`）；写入后必须重新读取锁表，通过标识确认这是自己刚写入的记录且没有被抢占，才允许切换角色执行。**切勿把自己刚写的锁误认为是别人占用的锁而导致卡死。**
- 任务说明允许修改范围与已有 `ACTIVE` 占用范围明显重叠时，后启动 flow 跳过该任务，避免多个 AI 同时改同一文件或目录。
- 完成、测试完成、返工完成、阻塞、验收收口或放弃领取后，当前角色必须释放对应占用。
- `ACTIVE` 占用超过 30 分钟无心跳时，不得直接抢占；后续 flow 只能把占用标记为 `STALE`，停止调度该任务，并交项目经理判断释放、恢复或改派。
- `IN_PROGRESS` 任务没有对应 `ACTIVE` 锁时，flow 不自动恢复，交项目经理判断是否释放、恢复或重派。

## 任务追溯

新任务的 `task.md` 必须写：

```markdown
## 来源

- 来源任务：
- 派生深度：
- 来源类型：
- 承接/替代：
- 上下文摘要：
```

- `TASK_BOARD.md` 只写短索引，不承载完整追溯。
- `TASK_BOARD.md` 每个当前任务必须有 `模块` 字段，模块定义见 `ai/TASK_MODULES.md`。
- 新任务必须填写 `主执行 Agent` 字段，用于复检和返工定位实际执行方；该字段不替代 `Owner`、`当前处理身份` 或 `下一步处理身份`。
- 项目经理定义或登记新任务前，必须按目标、模块、涉及文件、验收点、用户问题关键词和来源任务检索 `ai/TASK_BOARD.md`、`ai/TASK_ARCHIVE.md`、`.ai/tasks/`；必要时同步检查 `ai/TASK_LOCKS.md` 和 `ai/PM_DECISION_LIST.md`。
- 发现相同或高度相似任务时，不得直接新建；必须更新原任务、建立承接关系、关闭重复任务，或转 QA 拍板让用户确认。
- 单个任务直接下级最多 3 个。
- 任务派生深度最多 3 层。
- 达到第 3 层仍未解决时，项目经理关闭旧链并创建新基线任务。

## 测试与最终审查分流

- 临时修复仍走完整测试和最终审查，不合并、不跳过；只有 Owner 为 `aiskill` 的协作资产任务自检后直接归档。
- 测试角色完成任务测试后，测试通过进入 `REVIEW_READY`，测试失败进入 `TEST_FAILED` 并交项目经理裁决。
- 如果 `TASK_BOARD.md` 因混合行尾、编码异常或其他协作资产问题暂不可写，测试角色不得丢失测试结论：必须先把测试报告写入当前任务 `handoff.md` 或直接输出给用户，明确“建议状态：`REVIEW_READY` / `TEST_FAILED`”“下一步处理身份”和“任务板暂未写回原因”，再停止等待项目经理或 aiskill 处理协作资产问题。
- 项目经理评估 `TEST_FAILED`：决定原任务一次返工、新增承接任务、关闭任务、架构介入或进入 PM 决策。
- 测试不通过时，原任务最多允许一次 `FIX_REQUIRED` 返工；返工后再次测试不通过，不得继续原任务返工，必须新增承接任务或进入 PM 决策。
- 审查角色只领取 `REVIEW_READY`，完成最终审查后一律将原任务置为 `DONE`。
- 审查发现问题时，必须输出分类问题清单：`bug`、`风险`、`优化建议`。同类或同根因问题可以建议项目经理合并承接为一个任务。
- 审查发现 P0/P1 问题时，原任务仍按审查流程进入 `DONE`，但必须同步要求项目经理创建阻断承接任务或写入 `PM_DECISION_LIST.md` 的 `OPEN` 项，并明确受影响下游任务是否暂停；未完成阻断承接或 PM 决策前，关联任务不得继续执行。
- 审查报告若包含 P0/P1，必须填写 `承接要求：需要项目经理创建阻断承接任务 / 需要 PM 决策 / 已有承接任务 T-...`，并填写 `下游暂停：是/否`。审查角色不得为了填写尚未创建的任务 ID 而自行新建任务或臆造 ID。
- 项目经理创建阻断承接任务后，必须在新任务 `task.md`、新任务 `handoff.md`、来源任务 `flow-context.md` 或 PM 决策清单中记录承接任务 ID；审查报告可以追加“PM 承接结果”，但必须标明追加时间和角色，不强制回写原审查结论。
- 审查无问题时，记录“未发现对应级别问题”和未覆盖范围，并将原任务置为 `DONE`。
- 最终审查发现的问题不得打回原任务；项目经理根据问题清单新增承接任务、合并承接任务或进入 PM 决策。
- 最终审查发现问题后，原任务的职责到“产出审查报告和交接结论”为止；项目经理不得把原任务重新打开返工，必须先判断是否关闭原任务、创建承接任务或写入 PM 决策。原任务关闭后，后续修复只在承接任务中推进。
- 项目经理创建审查承接任务时，必须复用原任务已有需求澄清、验收标准、测试记录和审查报告作为输入；不要求用户重复填写关卡表。承接任务的 `task.md` 必须写明来源任务、问题批次、承接关系、允许范围、禁止范围、验收标准和“是否暂停下游”。
- 审查承接任务创建后进入自己的阶段流程：若问题边界已经由审查报告说清，可将 `需求澄清`、`验收标准` 记为已完成或替代完成，并把 `当前阶段` 设置到 `方案设计` / `开发实现` / `测试验证` 中最合适的下一关；若边界仍不清，当前阶段回到 `需求澄清`，当前处理身份为项目经理。
- flow 在原任务关闭后继续扫描任务板；发现审查承接任务时，不回到原任务，不要求用户再确认已明确的内容，而是按承接任务的 `当前阶段`、`阶段状态`、`当前处理身份` 调度对应维护人继续阶段流程。若承接任务后续到 `REVIEW_READY`，flow 可按受控审查规则切换到审查身份。
- 同一原始任务或同一问题链路累计新增承接任务最多 3 个；拟新增第 4 个承接任务时，必须写入 `ai/PM_DECISION_LIST.md` 交项目经理和用户拍板。
- 跨任务冲突、旧任务被覆盖、派生过深、需要重拆：交项目经理收口，不强行打回原任务。
- 架构判断问题：由项目经理交架构师给方案或接手。
- 协作资产 aiskill 任务：不进测试或审查队列。
- 审查角色完成最终审查时，必须在 `TASK_BOARD.md` 对应任务条目记录 `有效代码行`；缺少字段时由审查补齐。
- `有效代码行` 只统计业务源码的新增或实质修改行，排除空行、纯注释、纯格式化、文档、任务协作文件、生成文件和锁文件；统计优先级为目标 Git diff、可定位文件估算、`无法统计：原因`，均不阻断原任务进入 `DONE`。测试角色不统计有效代码行。

## PM 决策

- `PM_DECISION_LIST.md` 只保留 `OPEN` 和必要的短期 `DECIDED`。
- `OPEN` 决策默认局部阻塞：只阻塞关联任务，不阻塞其他任务或其他模块。
- 项目经理记录需要用户拍板的决策项后，若不是全局阻塞，交接中的 `下一步处理身份` 写具体被阻塞任务的下一角色或 `无`；`下一步动作` 写明“控制权交还 flow 继续扫描其他任务”。
- 决策项必须写 `阻塞范围`：`任务` / `模块` / `全局`。未填写时按 `任务` 处理。
- `阻塞范围：模块` 必须写明模块名；`阻塞范围：全局` 必须写明其他任务也不能继续的原因。
- 同一问题链路拟新增第 4 个承接任务、测试/审查结论与原始需求冲突、继续返工会制造循环时，必须进入 `PM_DECISION_LIST.md`。
- 已处理的 `DECIDED` / `CLOSED` 移到 `PM_DECISION_ARCHIVE.md`。
- 面向用户只输出 QA 拍板问题，不输出任务清单、任务链路、侦查过程或大表格。

## 归档分片

- `TASK_ARCHIVE.md` 或 `PM_DECISION_ARCHIVE.md` 超过 100 条时，由项目经理或 aiskill 新增按月分片归档任务；分片前不得临时批量移动历史记录。

## flow 状态灯

flow 面向用户只汇报固定模块状态表和状态灯，不列任务长清单：

```markdown
### 模块状态

| 模块 | 剩余 | 主要状态 | 可调度 |
| --- | --- | --- | --- |
| 模块名 | 数字 | 状态摘要 | 是/否/部分：短原因 |

### 优先模块选项

A. 选项名称（推荐）：一句原因
B. 选项名称：一句原因
C. 选项名称：一句原因

### 状态灯

摘要：一句用户可读的业务影响，不堆内部状态名。
执行中：有 / 无
占用中：有 / 无
待审查：有 / 无
待测试：有 / 无
审查阻断：禁止 / 明确强门禁时才有
待项目经理拍板：有 / 无
拍板阻塞范围：无 / 任务 / 模块 / 全局
阻塞：有 / 无
选项：A / B / C
下一步：等你选模块 / 切谁
```

只有存在多个模块时才输出 `### 优先模块选项`；当前只剩一个模块时，`下一步` 直接填写要切换的角色。

需要用户拍板时，只说“有项目经理拍板点”；只有当前候选任务被该拍板点阻塞、模块被阻塞或全局阻塞时，才切项目经理按 QA 提问。
用户指定模块优先级时，flow 按 `ai/TASK_MODULES.md` 和任务 `模块` 字段优先调度对应模块。
