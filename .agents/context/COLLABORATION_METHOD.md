# 协作方法与行为 P0/P1 硬规则

本文件承载四步协作法、精准修改和自检验证规则。入口文件 `AGENTS.md` 只保留指针。
本文件中的 P0/P1 是所有角色通用的行为硬规则，不是 code-reviewer 的代码审查问题分级，也不是 `ai/TASK_BOARD.md` 的任务优先级。

## 四步协作法

- 第一步：理解需求。用户提出需求后，AI 先返回理解，说明目标、边界、当前理解和可能影响范围。不确定的点必须逐条显式列出让用户确认，不默默填补。
- 第二步：问答澄清。AI 先排查可通过项目文件确认的信息，再提出关键问题。发现更简单的方案、明显风险或需求本身有问题时，必须主动提出，不顺从执行。
- 第三步：确认执行。AI 输出建议方案、影响范围、风险和回滚方式；用户确认后，才开始具体写入、修改、创建或删除。
- 第四步：验证。完成修改后必须先自检：改动是否符合原始请求、是否引入新问题、是否有遗漏。发现问题先修复再汇报，禁止拿未自检的结果直接汇报。"看起来对"不算，必须有可验证的证据。无法验证时必须明确告知用户。

涉及 `AGENTS.md`、`SKILL.md`、Prompt、业务代码、协议、token、设备密钥、Topic、gRPC、Swagger/proto 时，必须走完四步协作法。

flow 按已确认任务单自动调度执行时，不等于每个最小补丁都重新走完整 QA；若发现范围扩散、公共契约变化或用户拍板点，立即停回项目经理或用户确认。

机械一致性修复（任务 ID 路径、入口指针、错字、状态字段、混合行尾恢复为原行尾或项目约定行尾）可走轻量确认；结构性规则、Skill、Prompt 和业务代码仍走完整四步。

## 行为 P0：精准修改

- 每一行改动都必须能追溯到用户的原始请求，没有要求的不改。
- 禁止顺手改进相邻代码、注释、格式、命名风格。
- 禁止顺便重构没坏的东西。
- 发现无关的死代码只提不删，发现命名/风格问题只建议不改。
- 匹配现有代码风格，即便自己有偏好。
- 自己改动产生的孤儿代码要清理，但预先存在的不管。
- 准则：如果用户问“这行为什么改了”，答不上来就是不该改。

## 显式假设与主动质疑

- 第一步中把不确定的点逐条写出来，让用户确认，不默默填补。
- 第二步中发现更简单的方案、明显风险或需求本身有问题时，必须提出来，不顺从执行。

## 行为 P1：自检验证

- 完成修改后必须先自检，发现问题时先修复再汇报。
- 直接拿原始结论汇报但不修复 = 违规。
- 自检范围：改动是否符合用户原始请求、是否引入新问题、是否有遗漏。
- “看起来对”不算完成，必须有可验证的证据。
- 证据类型：测试通过、构建输出、运行时数据、文件内容确认。
- 如果无法提供验证证据，必须明确告知“无法验证”及原因。
