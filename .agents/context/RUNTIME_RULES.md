# AI Runtime 行为规则

本文件约束所有 AI 对话和所有身份的运行行为。角色 Skill 负责身份边界，本文件负责 THINK / ACT / VERIFY、上下文控制、工具优先级和防空转。

## 核心原则

- THINK 必须服务于 ACT；分析的目的必须是产生下一步行动。
- ACT 是安全、授权内、可验证的最小动作，不等于越权改代码。
- 不追求一次性获得绝对确定性；优先通过最小动作逐步推进任务。
- 任何角色都不得重复分析同一问题、无限方案比较或空转协商。
- 项目硬规则优先：无任务 ID 不改代码，敏感协议和密钥先确认，默认不运行 build/test/lint。

## ACT 定义

ACT 包括但不限于：

- 读取指定文件、任务板、任务说明、交接文件。
- 使用 `rg` 搜索关键词或文件。
- 查看 diff、状态、日志或错误输出。
- 在允许范围内做最小补丁。
- 执行用户已允许或任务明确要求的验证命令。
- 输出明确 action item、文件路径、命令、TODO 或阻塞原因。
- 更新授权范围内的任务状态、备注或交接记录。

以下不算 ACT：

- 重复说明“需要先理解”但不读取文件。
- 连续输出方案比较但不产生文件路径、命令、任务状态或补丁。
- 反复讨论风险但不查证据、不收敛到决策点。
- 以“信息还不够”为理由持续停留在纯分析。

## 防空转触发器

如果连续 2 次对用户输出、状态更新或可见执行步骤没有产生以下任一结果：

- tool call
- 文件读取或搜索结果
- 文件修改或 diff 输出
- 命令执行或测试结果
- API 调用或日志检查
- 明确的任务状态变化

必须立即输出：

```text
[AI Runtime]
Reasoning loop detected.
Force switching to ACTION mode.
```

随后进入 ACT，执行当前任务允许的最小可验证动作。

## 运行时状态机

每个任务执行过程必须处于以下运行时状态之一：

```text
PLAN -> ACT -> VERIFY -> RUNTIME_DONE
PLAN -> ACT -> VERIFY -> ACT
PLAN -> ACT -> BLOCKED
```

`RUNTIME_DONE` 只表示当前 AI 运行循环结束，不等同于任务板 `DONE`；任务板状态仍按 `ai/TASK_FLOW_RULES.md` 流转，Owner 为 `aiskill` 的协作资产任务例外。

禁止：

```text
PLAN -> PLAN -> PLAN
REVIEW -> REVIEW -> REVIEW
THINK -> THINK -> THINK
```

当状态无法继续推进时，必须转为 `BLOCKED`，写清阻塞原因、需要谁决策、建议动作和影响范围。

## Step Budget

- `MAX_VISIBLE_THINK_STEPS = 2`
- `MAX_VISIBLE_TOTAL_STEPS = 12`

可见步骤指对用户输出、工具调用、文件变更、命令执行、任务状态更新或交接记录更新。达到 THINK 预算后，必须选择当前最小可执行动作；达到总步数预算后，必须汇总当前状态，选择最可能推进任务的一步，进入 ACT 或 BLOCKED。

## Tool First

优先级：

1. 读取目标文件或任务说明。
2. `rg` 搜索相关符号、关键词、任务 ID。
3. 查看 diff、状态、日志或错误。
4. 在授权范围内做最小补丁。
5. 执行允许的验证命令。
6. 最后才继续推理。

禁止长时间脱离工具纯思考。无法使用工具时，必须说明原因，并输出下一步最小可执行动作。

## Windows 中文编码安全

- 读取协作资产首选 IDE 自带的 Read / Grep 工具；这类工具不经过终端编码层，能避免 PowerShell 默认 GBK 与 UTF-8 文件不匹配导致的乱码。
- 仅在确需 PowerShell / cmd 命令（管道、批处理、Git、系统操作）时才走 shell；走 shell 时必须显式设置 UTF-8 输出并使用 UTF-8 读取，例如先设置 `[Console]::OutputEncoding` 和 `$OutputEncoding` 为 UTF-8，再用 `Get-Content -Encoding utf8`。
- 固定片段见 `.agents/context/references/read-utf8.md`。
- 用户机器已配置 PS Profile 永久 UTF-8 时，规则仍按本段要求；不得假设环境已配置。
- 如果终端输出出现乱码，先按显示链路问题处理：重新用显式 UTF-8 读取或检查原始字节，禁止从乱码输出复制中文写回文件。
- 修改含中文文件前，必须确认文件正文未因读取链路损坏；无法确认时停止并询问用户。

## Context 熵控制

每次新对话、身份切换、从中断/压缩后恢复、或开始执行任务前，必须先检查当前对话上下文水位：

- 上下文水位低于 80%：继续按当前任务最小上下文执行，不额外读取历史。
- 上下文水位达到或超过 80%：先进入上下文压缩，不开始新的业务检查、代码修改或长推理。
- 压缩内容只保留当前用户请求、任务 ID、目标、允许/禁止范围、已改文件、关键结论、未完成动作、验证结果和阻塞点；删除无关历史、重复推理、旧截图解释和已关闭分支。
- 压缩后必须重新读取当前任务说明、当前交接和本 Runtime 规则，再进入 ACT。
- 如果当前工具无法主动压缩上下文，必须先输出可复制的压缩交接，并停止扩大分析；待用户压缩、开新线程或确认继续后再操作。

当文件过多、历史过长、上下文过深或任务链发散时，必须主动收敛：

- 只关注当前目标、当前报错、当前模块、当前任务文件。
- 丢弃与当前验收标准无关的信息。
- 不重新分析整个系统。
- 不把相邻优化合并进当前任务。
- 需要扩大范围时，先转 BLOCKED 或交项目经理拆任务。

## Retry 策略

允许失败，禁止因为害怕失败而持续思考。

- 不确定时先尝试最小方案。
- 失败后基于错误输出修正。
- 同一失败原因最多重试 2 次。
- 仍失败时转 BLOCKED，给出证据、已尝试动作和建议下一步。

## 高空转角色约束

- 架构师：2 个可见 THINK 步骤内必须给出文件结构、接口、实施步骤或进入 ACT；禁止空谈未来扩展性。
- 项目经理：内部必须收敛出当前最高优先级任务、下一执行动作和当前 blocker；面向用户时仍按项目经理 Skill 的 QA 拍板格式输出。
- flow：每轮扫描后只能调度、归属报告或 BLOCKED，禁止无动作循环扫描。
- 测试：必须输出测试通过/不通过、验收点覆盖、验证命令或未运行原因；禁止扩展成最终代码审查。
- 审查：必须输出真实问题，或明确“未发现对应级别问题”和未覆盖范围；禁止为了凑结论反复审查。
- 前端/后端：不得长时间“理解代码库”；读取必要上下文后必须最小补丁、验证或 BLOCKED。

## 验证要求

每次 ACT 后必须进入 VERIFY：

- 有修改：复查目标文件内容或 diff。
- 有命令：记录命令和结果摘要。
- 未运行验证：说明未运行原因。
- 无法验证：明确写出无法验证的原因和剩余风险。
